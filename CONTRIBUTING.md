# Руководство по участию в проекте

Спасибо за интерес к проекту! Это руководство поможет эффективно работать с SRD-структурой.

## Принципы участия

### 1. Минимализм
- Создавайте только необходимые документы
- Избегайте дублирования информации
- Удаляйте устаревший контент

### 2. Практичность
- Фокусируйтесь на применимых знаниях
- Приводите рабочие примеры
- Проверяйте актуальность информации

### 3. Системность
- Размещайте документы в правильных ячейках матрицы
- Поддерживайте связи между документами
- Следуйте принципам FPF

### 4. Ясность
- Пишите понятно и структурировано
- Используйте русский язык
- Избегайте жаргона без объяснений

## Структура SRD

Перед добавлением контента определите:

1. **Система** — к какой системе относится документ?
   - `0.Management` — управление проектом
   - `1.Supersystem` — внешний контекст
   - `2.TargetSystem` — целевой продукт
   - `3.CreationSystem` — процесс создания

2. **Роль** — какой аспект описывается?
   - `X.1.Meaning` — смысл, цели, требования
   - `X.2.Architecture` — структура, компоненты
   - `X.3.Operations` — процессы, действия

3. **Путь** — `content/{N}.{System}/{N}.{M}.{Role}/document.md`

## Шаблон документа

Каждый документ в `content/` должен иметь YAML frontmatter:

```markdown
---
system: "TargetSystem"
role: "Architecture"
title: "Название документа"
date: "YYYY-MM-DD"
author: "Ваше имя (опционально)"
tags: [tag1, tag2, tag3]
status: "draft"              # draft, review, stable
related:
  - ../path/to/related.md
fpf_principles:
  - principle-key
---

# Название документа

## Контекст

Зачем нужен этот документ?

## Содержание

Основная информация...

## Связи

- Связан с [документ](../path/to/doc.md)
- Зависит от [принцип](.fpf/INDEX.md#principle)

## Примеры

Рабочие примеры использования...
```

## Процесс добавления контента

### 1. Планирование

- [ ] Определите систему и роль
- [ ] Проверьте, нет ли похожего документа
- [ ] Изучите связанные документы
- [ ] Проверьте релевантные принципы в `.fpf/INDEX.md`

### 2. Создание

- [ ] Создайте файл в правильной папке
- [ ] Добавьте frontmatter по шаблону
- [ ] Напишите содержание
- [ ] Добавьте примеры (если применимо)
- [ ] Укажите связи с другими документами

### 3. Проверка

- [ ] Frontmatter корректен и полон
- [ ] Документ находится в правильной папке
- [ ] Связанные документы существуют
- [ ] Принципы FPF соблюдены
- [ ] Нет дублирования информации

### 4. Публикация

- [ ] Обновите `related` в связанных документах
- [ ] Создайте Pull Request с описанием
- [ ] Дождитесь review
- [ ] Внесите правки по feedback

## Работа с FPF

### Когда добавлять принцип в `.fpf/INDEX.md`:

✅ **Добавляйте, если:**
- Принцип фундаментален для проекта
- Принцип повторно применяется
- Принцип влияет на архитектурные решения
- Принцип требует последовательного соблюдения

❌ **Не добавляйте, если:**
- Это частная реализационная деталь
- Принцип специфичен для одного документа
- Принцип очевиден из контекста
- Принцип временный или экспериментальный

### Формат принципа в FPF:

```markdown
### principle-key

**Описание:** Краткое описание принципа

**Обоснование:** Почему этот принцип важен

**Применение:**
- Контекст 1: как применять
- Контекст 2: как применять

**Примеры:**
- [Документ 1](../content/path/to/doc.md)
- [Документ 2](../content/path/to/doc.md)
```

## Типы изменений

### Документация
- Новый документ в `content/`
- Обновление существующего документа
- Исправление ошибок в документации

### Структура

- Добавление новых категорий

### Принципы
- Добавление принципа в `.fpf/INDEX.md`
- Обновление существующего принципа
- Удаление устаревшего принципа

### Шаблоны
- Улучшение шаблонов документов
- Добавление новых шаблонов
- Обновление frontmatter схемы

## Стиль документации

### Язык
- **Основной:** русский
- **Технические термины:** можно на английском с пояснением
- **Код и примеры:** комментарии на русском

### Форматирование
```markdown
# Заголовок 1 (один на документ)
## Заголовок 2 (основные разделы)
### Заголовок 3 (подразделы)

**Жирный** для важных терминов
*Курсив* для акцента
`код` для команд и имён

- Маркированные списки
1. Нумерованные списки
```

### Ссылки
- Относительные для внутренних документов: `[текст](../path/to/doc.md)`
- Абсолютные для внешних ресурсов: `[текст](https://example.com)`
- Якоря для секций: `[текст](#section-name)`

## Именование файлов

### Правила
- Используйте kebab-case: `my-document.md`
- Расширение: `.md` для Markdown
- Язык: русский или английский (последовательно)
- Краткость: до 50 символов

### Примеры
✅ Хорошо:
- `api-architecture.md`
- `user-requirements.md`
- `deployment-process.md`

❌ Плохо:
- `Архитектура API.md` (пробелы и кириллица)
- `doc1.md` (неинформативно)
- `very-long-document-name-that-describes-everything.md` (слишком длинно)

## Review процесс

### Критерии приёмки

**Обязательно:**
- [ ] Документ в правильной папке SRD
- [ ] Валидный YAML frontmatter
- [ ] Нет дублирования существующей информации
- [ ] Связи с релевантными документами
- [ ] Соответствие принципам FPF

**Желательно:**
- [ ] Рабочие примеры
- [ ] Визуализация (диаграммы, схемы)
- [ ] Список ссылок на дополнительные материалы

### Роли в review

**Автор:**
- Создаёт качественный контент
- Отвечает на вопросы reviewer
- Вносит правки по feedback

**Reviewer:**
- Проверяет соответствие guidelines
- Даёт конструктивную обратную связь
- Следит за целостностью структуры

**Maintainer:**
- Финальное одобрение
- Merge Pull Request
- Поддержка актуальности `.fpf/INDEX.md`

## Частые вопросы

### Куда добавить документ о процессе разработки?
`3.CreationSystem/3.3.Operations/` — операции системы создания

### Куда добавить описание требований пользователей?
`2.TargetSystem/2.1.Meaning/` — смысл целевой системы

### Куда добавить оргструктуру команды?
`0.Management/0.2.Architecture/` — архитектура управления

### Куда добавить описание взаимодействия с внешними системами?
`1.Supersystem/1.3.Operations/` — операции надсистемы

### Как связать документы из разных ячеек?
Используйте поле `related` в frontmatter обоих документов

### Когда создавать новую папку в content/?
Только если нужна новая система или роль (редко), обсудите с maintainer
